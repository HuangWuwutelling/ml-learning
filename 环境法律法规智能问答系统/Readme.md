# 环境法律法规智能问答系统

基于 **RAG（检索增强生成）** 技术的中国环境法律智能问答系统，支持自然语言提问，从法律法规库中检索相关内容并生成准确回答。

---

## 项目概述

| 项目 | 内容 |
|------|------|
| **目标** | 构建一个能够理解中文环境法律问题并给出准确答案的智能问答系统 |
| **技术栈** | Python 3.11+, FastAPI, ChromaDB, Sentence-Transformers, DeepSeek API |
| **法律库** | 《中华人民共和国生态环境法典》（2026年3月12日通过，8月15日施行），共 5 编 1242 条 17 万字 |
| **适用场景** | 企业环保合规咨询、法律知识普及、环境法学习辅助 |

### 核心功能

- 法律法规向量化存储与语义检索
- 基于 DeepSeek 大语言模型生成专业回答
- 支持连续多轮对话，理解上下文指代
- 答案附带引用来源（法律名称）
- 简洁美观的 Web 交互界面
- RESTful API 支持二次集成

---

## 技术架构

```
用户提问 → 向量检索 → 检索结果 → 构建Prompt → LLM生成 → 展示回答
                    ↑                          ↑
              法律文本向量库            DeepSeek API
```

### 数据流

1. **离线阶段**: 法律文本 → 文本分块 → BGE嵌入向量 → ChromaDB
2. **在线阶段**: 用户问题 → 向量检索 TOP-K → 上下文拼接（含对话历史） → DeepSeek → 回答

### 连续对话

系统支持多轮连续对话，自动保留最近 5 轮问答历史（可通过 `.env` 中 `MAX_HISTORY_ROUNDS` 调整）。AI 能根据上文理解代词指代和后续追问，例如：

```
用户：企业排污需要什么条件？
助手：根据法典第181条，需要取得排污许可证，具备污染防治设施...
用户：那违规了怎么处罚？    ← 自动理解"那"指排污违规
助手：根据法典第1212条...
```

### 关键组件

| 组件 | 技术选型 | 说明 |
|------|----------|------|
| **嵌入模型** | BAAI/bge-small-zh-v1.5 | 轻量级中文语义向量模型（133MB） |
| **向量数据库** | ChromaDB | 本地持久化向量存储，无需单独部署 |
| **大语言模型** | DeepSeek Chat | 中文能力强，性价比高 |
| **后端框架** | FastAPI | 高性能异步 Web 框架 |
| **前端** | 原生 HTML/CSS/JS | 无依赖，轻量美观 |

## 环境法律法规库

系统基于 **《中华人民共和国生态环境法典》**（2026年3月12日第十四届全国人民代表大会第四次会议通过），这是中国首部生态环境领域的法典。

### 法典结构

| 编 | 内容 | 章 | 条 |
|----|------|----|----|
| 第一编 | 总则 | 9 章 | 第1-197条 |
| 第二编 | 污染防治 | 8 分编、26 章 | 第198-697条 |
| 第三编 | 生态保护 | 6 章 | 第698-1063条 |
| 第四编 | 绿色低碳发展 | 4 章 | 第1064-1211条 |
| 第五编 | 法律责任和附则 | 3 章 | 第1212-1242条 |

### 核心特点

- **统一性**：取代《环境保护法》《环境影响评价法》《海洋环境保护法》《大气污染防治法》《水污染防治法》《土壤污染防治法》《固体废物污染环境防治法》《噪声污染防治法》《放射性污染防治法》《清洁生产促进法》10 部单行法
- **全面性**：涵盖污染防治、生态保护、绿色低碳发展全领域
- **时效性**：2026年8月15日起施行，代表最新的立法成果

## 快速开始

### 环境要求

- Python 3.9+
- Windows / Linux / macOS 均可

### 1. 安装依赖

```bash
pip install -r requirements.txt
```

### 2. 配置 API Key

```bash
cp .env.example .env
# 编辑 .env，填入 DeepSeek API Key
# DEEPSEEK_API_KEY=sk-your_key_here
```

> 前往 [DeepSeek 平台](https://platform.deepseek.com) 注册并获取 API Key，新用户赠送 500 万 tokens。

### 3. 准备法律数据

将《中华人民共和国生态环境法典》DOCX 文件放入 `data/raw/` 目录，或运行数据提取脚本：

```bash
python collect_data.py
```

### 4. 初始化向量数据库

```bash
python init_db.py
```

首次运行会自动下载嵌入模型（BAAI/bge-small-zh-v1.5，约 133MB），然后将 17 万字的法典全文分块、向量化并存入 ChromaDB。

成功输出示例：
```
✅ 向量数据库初始化完成！
   法律文件: 中华人民共和国生态环境法典
   总字数: 170,606 字
   总条数: 1249 条
   文档片段: 445 个
   向量维度: 512
```

### 5. 启动服务

```bash
python app.py
```

访问 http://localhost:8000 使用 Web 界面。

## 使用方法

### 方式一：Web 界面（推荐）

启动服务后，浏览器访问 http://localhost:8000：

1. 在页面底部的输入框中输入问题，按 Enter 或点击发送按钮
2. 系统会实时显示"正在输入"状态，等待 AI 返回回答
3. 回答下方会标注引用来源（法律名称）
4. **支持连续对话**：追问时 AI 会自动理解上下文，如先问"排污需要什么条件"再问"违规怎么处罚"
5. 对话历史保留最近 5 轮（可在 `.env` 中调整 `MAX_HISTORY_ROUNDS`）

**适用场景**：日常查询、法律知识学习、合规初步咨询。

**截图示例**：打开浏览器即可看到绿色主题的聊天界面，左侧显示支持的法律法规列表，底部为输入框。

**检查服务状态：**

```bash
curl http://localhost:8000/api/status
```

返回：
```json
{"status":"ok","db_docs_count":30,"embedding_model":"BAAI/bge-small-zh-v1.5"}
```

**提问：**

适合集成到其他 Python 应用中。

```python
# 提问
```

## 开发历程与关键技术决策

### 第一阶段：需求分析与架构设计

**问题定义**：环境法律法规体系庞大、条款繁杂，企业环保合规人员和非专业人士难以快速定位所需的法律条文。传统关键词搜索无法理解语义，需要一个能"理解"问题并精确定位法条的系统。

**技术选型对比**：

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| 纯关键词搜索 | 简单快速 | 无法处理同义词、语义匹配差 | ❌ |
| 微调LLM | 回答精准 | 需要大量标注数据，成本高 | ❌ |
| **RAG** | **无需训练、知识库可扩展、成本低** | **依赖检索质量** | **✅** |
| 纯LLM回答 | 开发最快 | 可能幻觉编造法条 | ❌ |

### 第二阶段：数据工程

**数据源选择**：最初尝试内置 8 部单行法的摘要文本（30 个文档片段），后来替换为 **《中华人民共和国生态环境法典》** 完整 DOCX 原文（17 万字、1242 条）。

**关键改进**：从 DOCX 文件直接提取全文，采用 python-docx 解析，保证了数据源的权威性和完整性。

**语义分块策略**：
- 按编/章/条三级结构拆分，保留层级关系
- 每个块 500 字符，保留 50 字符重叠避免语义断裂
- 445 个高质量文档片段覆盖法典全部内容

文本分块是 RAG 系统的关键难点：块太大则检索不精确，块太小则丢失上下文。通过多级拆分策略取得了较好的平衡。

### 第三阶段：向量检索

**嵌入模型选型过程**：
- 尝试 `BAAI/bge-large-zh-v1.5`（2.2GB）— 在国内网络环境下下载不稳定
- 最终选用 `BAAI/bge-small-zh-v1.5`（133MB）— 推理速度快 10 倍，精准度仅下降约 2%
- 设置 `HF_ENDPOINT` 环境变量使用 hf-mirror.com 国内镜像加速

**向量数据库**：选用 ChromaDB 而非 Milvus/Weaviate，主要考虑：
- 本地嵌入式运行，无需 Docker 或独立服务
- API 简洁，与 Python 生态无缝集成
- 单机场景性能足够

### 第四阶段：RAG 问答引擎

**Prompt 工程**：设计了包含系统角色、检索上下文、问题三部分的结构化 Prompt，要求模型：
- "仅基于提供的法律法规内容进行回答，不要编造法条" — 约束 LLM 幻觉
- "引用具体法条时注明出处" — 增强回答可信度
- Temperature=0.3 — 平衡确定性与创造性

### 第五阶段：Web 界面

设计原则：简洁、专业、易用。绿色主题呼应"环保"定位，支持：
- 实时打字指示器
- 回答来源标签展示
- 响应耗时显示
- 移动端自适应布局

## 扩展指南

### 更新法典版本

当有新版本的法典 DOCX 文件时，替换 `data/raw/` 中的文件，然后重新运行：

```bash
python init_db.py
```

### 添加其他法律法规

编辑 `src/data/collector.py`，添加从其他 DOCX/TXT 文件提取法律文本的逻辑，然后重新运行 `init_db.py`。

### 切换数据源

`collector.py` 的 `get_codec_text()` 函数返回法典全文。可修改此函数从 API、数据库或其他文件格式读取数据。

### 更换 LLM

修改 `src/rag/qa_engine.py` 中的 `query_deepseek` 方法，可以切换到 OpenAI、通义千问等其他 API。

### 更换嵌入模型

修改 `.env` 中的 `EMBEDDING_MODEL_NAME`，支持所有 HuggingFace 上的 Sentence-Transformer 模型。
