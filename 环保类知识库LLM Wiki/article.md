# 从RAG到LLM Wiki：用AI构建生态环境法典知识库

> **摘要**：上篇文章用 RAG 做了个生态环境法典智能问答系统，效果不错但知识无法积累。本文分享如何用 LLM Wiki 模式重新构建知识库——通过有道云笔记 + AI Agent，将 16.8 万字的法典原文编译为 12 篇结构化的概念页面，实现知识的持续积累和交叉引用。文章详细对比了 RAG 与 LLM Wiki 两种范式的差异，并记录了完整的实操过程。

这是上一篇公众号文章《我的第一个 RAG 项目：生态环境法典智能问答》的后续。感兴趣可以先回去看看第一篇，再来看这篇。

上回用 RAG（检索增强生成）做了一个生态环境法典智能问答系统，效果还行。但用了一段时间后，我发现了几个问题：

- 每次问答都要重新检索，同样的知识反复查
- 无法对知识进行整理、关联和积累
- 法条之间有什么关系，系统不知道
- 查过的好问答没法保存下来复用

简单说就是：**用完即弃，知识不增长**。

后来了解到 Andrej Karpathy 提出的 **LLM Wiki 模式**——把知识编译一次、持续维护，而不是每次查询从零开始检索。于是我开始尝试用这个思路重新构建我的生态环境法典知识库。

---

## 什么是 LLM Wiki？

LLM Wiki 是 Andrej Karpathy（OpenAI 联合创始人，前 Tesla AI 总监）在 2025 年底提出的一种知识管理范式。

传统的 RAG 像给 AI 配了一本词典——每次问问题，AI 都去翻词典找答案。问题是，这本词典是**死的**：你查 100 次，词典还是原来的词典，不会因为你查过就变得更聪明。

而 LLM Wiki 的思路是：**把知识真正"存进去"**。

具体来说，就是把原始素材经过 AI 处理后，编译成结构化的"Wiki 页面"——每页讲一个概念、一个实体或一个制度。页面之间有交叉引用，矛盾和空白会被标记。这样知识是**活的**，每次摄入新内容，整个知识库都会随之生长。

两种模式的对比如下：

| 维度 | RAG | LLM Wiki |
|------|-----|----------|
| 知识存储 | 向量数据库（语义索引） | 结构化的 Wiki 页面 |
| 每次查询 | 重新检索全部文档 | 直接读取相关页面 |
| 知识增长 | 不增长（查询不影响知识库） | 每次摄入都在积累 |
| 交叉引用 | 不存在 | 页面间有 → 标记 |
| 矛盾检测 | 不检测 | 标记矛盾，人工确认 |
| 可读性 | 机器读的向量 | 人也能直接读 |

## 选型：有道云笔记 + YoudaoNote LLM Wiki

选工具的时候，核心诉求就两个：

1. **云端存储**——知识库不能只存在本地，要能在手机、平板、电脑上随时查看
2. **AI Agent 可操作**——AI 能帮我一键摄入素材、创建页面、维护索引

最终选用的是**有道云笔记 + YoudaoNote LLM Wiki Skill** 的组合。

有道云笔记是国内成熟的云笔记产品，API 完善，而 YoudaoNote LLM Wiki 是一个开源的 AI Agent 技能包，专门为 Claude Code 设计，可以让 AI 自动完成 Wiki 的创建、摄入、查询和维护。数据存在有道云，查看和管理用有道云 App，AI 通过 CLI 操作云端笔记。

## 安装过程

安装分为三步，都不复杂。

### 第一步：安装有道云笔记 CLI

到有道云笔记官网下载 CLI 工具，安装好后在终端验证：

```bash
youdaonote list
```

### 第二步：申请 API Key

在有道云笔记的 MCP 平台申请 API Key，配置到环境变量中。这一步是为了让 CLI 有权限操作你的云笔记。

### 第三步：安装 YoudaoNote Wiki Skill

我用的是 Claude Code，下载技能包后解压到 skills 目录：

```bash
~/.claude/skills/youdaonote-llm-wiki/
```

然后在 Claude Code 中就可以直接使用这个技能了。

## 实战：构建 eco-law-wiki 知识库

安装完成后，告诉 Claude：

> 帮我创建一个知识库，主题是我国生态环境的法律法规、政策规范

AI 会自动创建文件夹结构、schema 文件和索引，整个过程不到一分钟。知识库命名为 `eco-law-wiki`，目录结构如下：

```
eco-law-wiki/
├── schema.md          ← 结构约定、领域定义
├── index.md           ← 内容目录
├── log.md             ← 操作日志（只追加）
├── raw/               ← 原始素材（只读）
├── entities/          ← 实体页（部门、机构）
├── concepts/          ← 概念页（制度、机制）
├── comparisons/       ← 对比分析页
└── queries/           ← 查询归档页
```

### 摄入过程

获取《生态环境法典》的原文。需要说明的是，直接从生态环境部官网抓取内容效果不太理想，因为官方网站是动态页面。所以下载了 DOCX 格式的官方原文，让 AI 转换成 Markdown。

AI 提取了全部 1242 条、约 16.8 万字的内容，结构化为标准 Markdown 文件，存入 `raw/` 目录作为不可修改的原始素材。

然后 AI 自动分析法典内容，识别关键制度和概念，逐一创建 Wiki 页面：

- **核心条目**：生态环境法典概述、结构、历史意义
- **制度概念**：分区管控制度（"三线一单"）、排污许可管理、生态环境保护督察、应对气候变化
- **新增概念**：绿色低碳发展、生态保护、突发生态环境事件应对、信息公开与公众参与、生态保护补偿、生态环境损害赔偿
- **实体页**：中华人民共和国生态环境部
- **废止法律**：被法典取代的 10 部旧法律

每个页面都标注了具体的法律条款编号，页面之间用 `→` 标记交叉引用。比如"排污许可管理"页面里会引用"废止的 10 部环保法律"。

### 操作日志

最让我满意的是 `log.md`——每次操作的完整记录，AI 都会自动追加到日志中。从初始化到现在做了什么，一清二楚。

截至 2026 年 4 月 29 日，知识库的规模是：

| 维度 | 数值 |
|------|------|
| 原始素材 | 3 篇（总览 + 第一编 + 全文） |
| 概念页 | 12 篇 |
| 实体页 | 1 篇 |
| 总笔记数 | 16 篇 |

## 效果对比：RAG vs LLM Wiki

为了直观对比，我拿同一个问题来测试：

### 问：企业排污需要什么条件？

**RAG 的回答方式**：

系统去向量数据库里检索最相关的几个文档片段（可能是第 174 条、第 180 条的一部分），然后把这些片段拼接给大模型生成回答。每次都要重新检索、重新组合，不可控因素多——有时候检索到的片段刚好跨了章节，语义就不连贯。

**LLM Wiki 的回答方式**：

AI 直接从概念页 "排污许可管理" 中读取完整的制度介绍。这个页面是经过整理和结构化的——什么条件下需要许可证、许可证怎么申请、后续有什么义务，都写得清清楚楚。回答直接从页面内容里组织，稳定且完整。

### 更重要的区别：知识积累

RAG 模式下，我查 100 次"排污许可"，知识库也不会因此变得更好。但在 Wiki 模式下，每次摄入新素材——比如今天刚发布的《农田灌溉水质标准》——AI 会自动更新相关概念页，所有参考文献和交叉引用随之更新。知识库是**在生长**的。

## 一些感受

从 RAG 到 LLM Wiki，本质上是从"让 AI 临时翻书"到"让 AI 真正理解并整理知识"的转变。前者适合一次性问答，后者适合长期积累。

它不是要取代 RAG，而是提供了一个不同的视角：如果你的目标不只是问答，而是**真正构建一个可积累、可维护的知识体系**，那么 Wiki 模式是更好的选择。

当然，这个模式也有局限性。它依赖 AI Agent 的质量——如果 AI 总结错了或者忽略了重要内容，知识库的质量就会打折扣。另外，对于需要快速检索大量信息做即时判断的场景，RAG 仍然更合适。

但作为一个持续更新的个人知识库，LLM Wiki 这个方向我觉得是对的。后续我还会继续往 eco-law-wiki 里补充更多素材，包括环境质量标准、排放标准等 PDF 文件。

## 怎么分享给别人？

有人问：这个知识库存在有道云笔记里，是不是只能自己用，能不能分享给别人问答？

有三种方式：

**方式一：直接分享笔记** — 有道云笔记支持分享单个笔记或整个文件夹，生成链接发给别人就能在浏览器看，但不能问答。

**方式二：建一个公开问答界面** — 把 Wiki 页面内容作为知识源，搭一个对话页面（比如用上一篇 RAG 项目的前端框架），别人打开就能问。相比纯 RAG 方案，这里的知识源是已经整理好的概念页，检索质量更高。

**方式三：导出为静态站点** — 用 MkDocs、VuePress 等工具把 Wiki 页面生成文档站，部署到 GitHub Pages 或 Vercel 上，方便浏览。

我个人倾向方式二——把两个项目结合起来：用 RAG 项目的前端，配上 Wiki 项目整理好的概念页作为知识源，既保留了知识的积累，又支持分享问答。

如果你也在做类似的知识管理项目，欢迎交流。
